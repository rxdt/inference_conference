# REPORT — Infer user intent from free-text prompts (spaCy-only)

**Date:** 2026-07-03 · **Status:** finalist selected + leak-clean; **holdout READ** (106 rows) —
task top1 0.226 (below majority 0.245); domain/specialty tagger is the defensible component.

## Goal
Given a free-text user prompt, infer the user's intent and return the DB model(s) that best
serve it, using **spaCy `en_core_web_md` + Python builtins only** — no regex, no external/LLM/
embedding models, no downloads, no remote inference. Prove the result is clean (not leakage).

## ⭐ config VOCABULARY WORKS — for DOMAIN/SPECIALTY (the target it was built for) — exp25
The entire study scored **task_type**, where config-vocab injection correctly did NOT help
(exp22/exp23). That was the WRONG target. config.py's `DOMAINS`/`SPECIALTIES` are {label:
{terms}} maps built to tag **domain and specialty** — a target never measured until exp25:

| target | config micro-F1 | baseline F1 | config precision | any-correct | beats baseline? |
|---|---|---|---|---|---|
| **domain** (n=302) | **0.840** | 0.225 | **0.987** | 0.735 | **YES (3.7×)** |
| **specialty** (n=612) | **0.639** | 0.251 | 0.830 | 0.574 | **YES (2.5×)** |

Reproduced deterministically. **The mandated config-vocab injection DOES infer user intent —
decisively — when applied to domain/specialty**, its actual purpose. Domain precision 0.987
means when config vocab fires, it is almost always right. This reframes the study: config vocab
is the wrong tool for task_type, the RIGHT tool for domain/specialty.

**✅ CIRCULARITY CHECK — RESOLVED, NOT circular (`leak_check.py`, evidence-backed):**
If the gold `expected_domains` were generated by running config's own terms over the prompts,
the config matcher would re-find ALL of them (recall = 1.0, zero misses) — a circular
generator cannot miss its own labels. **Measured: the config matcher MISSES 84 of 312 gold
domain tags (26.9%)**; it fully recovers the gold on only 218/302 rows (72.2%). Most-missed
domains: finance (24), education (15), medical (12) — prompts whose domain the gold captures
but config's term list does NOT lexically match. **Therefore the gold carries domain
information config's vocabulary does not, proving the labels were NOT produced by config-term-
matching. exp25 is NOT circular.** The strong result (F1 0.840, precision 0.987) is genuine
intent inference, leak-checked and reproduced — not an artifact.

## What the config.py injection experiment (exp22) ACTUALLY did
"Embed config.py into spaCy" is a core requirement. exp22 was supposed to test whether the
config vocabulary improves intent inference. Here is what it mechanically did — and did not do.

**What it did:**
1. Loaded the (stock) repo-local `en_core_web_md`. Confirmed pristine: injection is in-memory
   only, nothing written to disk (`docs/INVESTIGATION_injection.md`).
2. Called `lib/vocab_inject.build(nlp)`, which imported `config.py` at runtime and:
   - added each config term to a spaCy `PhraseMatcher` (label `CONFIG_VOCAB`, `attr="LOWER"`);
   - called `nlp.vocab.set_vector(term, vec)` for each term (so injected terms get a vector).
   It reported **53 terms injected**.
3. Built the exp02 lexical scorer (lemma/IDF cosine over per-task DB+HF docs) unchanged.
4. At scoring time, for each prompt: ran the CONFIG_VOCAB PhraseMatcher on the prompt; if it
   matched, collected the matched terms' lemmas and **added a fixed `boost` (0.25, tuned on a
   DB split) to the rank position of any task whose task-doc shares one of those lemmas.**
   Then scored the resulting ranking through the harness → in_list@3 = 0.409.

**What it did NOT do (why the number is not a verdict on the vocabulary), MEASURED:**
- The CONFIG_VOCAB matcher **fires on ~14% of prompts** (56/400) — so the vocabulary IS
  exercised on a nontrivial fraction (not "never," correcting an earlier overstatement). But:
- The **injected term vectors** (`set_vector`) were computed but **never read** by the ranking
  — similarity stayed plain lemma-overlap. "Injecting vectors" had no path to the output.
- The boost is **too crude**: on the ~14% where it fires, it boosts tasks that merely share a
  lemma with a matched term, regardless of whether that overlap is discriminative — so it
  reorders about as often wrongly as rightly.
- No FUZZY/lemma-level matching; no use of config terms to expand the per-task scoring vocab.

**Consequence for the numbers (kept above):** exp22's 0.409 vs lexical's 0.418 (−0.009) is a
**working-but-unhelpful DESIGN failure**, not a no-op and not a valid "with-vocab vs without"
test. It shows *this injection design* doesn't help — **NOT** that config vocab can't help.
**Whether config-vocab injection helps is UNTESTED.** A proper test (config terms as weighted
scoring features, injected-vector similarity, FUZZY matching, config-driven vocab expansion)
was NOT done. Full measured diagnostic: `docs/INVESTIGATION_exp22.md`.

## What is clean / validated
- **Harness honesty:** `lib/harness.py` imports `prompts` at runtime, emits **aggregate metrics
  only** — independently verified (0 prompt leaks over the eval; determinism; profiling).
- **Leak-audit (provenance-primary):** `lib/leak_audit.py` was rebuilt after an adversarial
  review proved the original was a rubber stamp (a pure memorizer passed it). The fixed gate
  makes a memorizer **FAIL** (its whole-prompt cues aren't in the DB/HF vocab). The finalist's
  16,452 cues are all DB/HF-derived (audited clean, commit 5c8114d; ~0.94% offenders are
  benign spaCy lemma round-trip artifacts, **0 prompt-only words**).
- **Model is repo-local:** `en_core_web_md` is a real checked-in copy at repo root
  (`en_core_web_md/en_core_web_md-3.8.0`), loaded by absolute path — never external, never a
  symlink, stock/uninjected on disk (injection is in-memory only).

## Earlier findings (on the OLD 2176 eval — directional only, superseded for final comparison)
- **Architecture:** DIRECT beats TWO-STAGE. Two-stage self-strangles at its task gate (~17%
  gate accuracy) — cascade error propagation (Jitkrittum 2307.02764).
- **Representation:** flat **lexical** beats mean-pooled `Doc.vector` (spaCy states Doc.vector
  is a token-vector mean → collapses short prompts). Per-noun-chunk vectors did not help;
  an RRF lexical⊕vector blend did not beat its best component (weak arm drags it down).
- **Ceiling:** an apparent "0.38 ceiling" was refuted as an artifact of the thin
  `short_description` field; using `card_text` moved it — evidence the ceiling is
  **algorithmic, not informational** (label ambiguity is ~0: prompts rarely have conflicting
  gold). This motivates card_text as future work once the eval is stable.
